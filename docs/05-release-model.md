# リリースモデルとバージョン戦略（Kotlin / Java）

> 「Kotlin に LTS はある？」という疑問から調べた内容のメモ。
> このプロジェクトのバージョン選定（Java 25 LTS / Kotlin 2.4 最新）の根拠でもある。

## 結論

- **Java には LTS の概念がある。Kotlin には無い。**
- そのため戦略が逆になる:
  - Java → **LTS 版に留まる**のが定石
  - Kotlin → **常に最新に追従する**のが定石

## Java（LTS の概念アリ）

「LTS 版」と「通常版」がハッキリ分かれている。

| 種別 | 内容 |
|------|------|
| LTS 版 | 2〜3 年ごと（**8 → 11 → 17 → 21 → 25**）。数年間の長期サポート付き |
| 通常版 | その間の半年ごとの版（18, 19, 20, 22, 23, 24…）。サポートは**約半年で終了** |

→ 企業は「LTS だけ使う」運用が一般的。
→ 本プロジェクトで **Java 25 (LTS)** を選んだのはこのため。

## Kotlin（LTS の概念ナシ）

LTS という区分は存在しない。代わりに以下のモデル。

| 項目 | 方式 |
|------|------|
| 言語リリース | **半年ごと**（2.0 → 2.1 → 2.2 → 2.3 → 2.4…） |
| ツール / バグ修正リリース | 3 ヶ月ごと・随時 |
| サポート対象 | **バグ修正・セキュリティ修正は「最新版のみ」** |
| 古い版のサポート | 基本無し（常に最新へ上げるのが推奨） |

### LTS が無い代わりの安心材料 = 強い後方互換性

- Kotlin 1.0 時代のコードが今の 2.x でも基本そのまま動く。
  → 「最新に上げたら壊れた」が起きにくい設計。
- **言語バージョン**の仕組みで、新しいコンパイラを使いつつ言語仕様を古い版に固定も可能
  （`-language-version` オプション）。

## 対比表

| | Java | Kotlin |
|---|------|--------|
| LTS の概念 | **アリ**（8/11/17/21/25） | **ナシ** |
| サポート戦略 | LTS を長期サポート | 最新版のみ修正・常時追従 |
| 安心の担保 | LTS に留まる | 強い後方互換性で上げやすい |

→ 本プロジェクトの **Java=LTS の 25 / Kotlin=最新の 2.4** という選択は、
　それぞれの流儀に沿った定石。
