# 用語集・概念整理

> 学習の過程で「混同しやすい用語」や「役割の違い」を整理していくメモ。

## 各ツールの役割（レイヤーで整理）

混同しやすいので、それぞれが「何をする係」かをハッキリさせる。

| 名前 | 正体 | 役割 |
|------|------|------|
| PostgreSQL | データベース本体 | データを保存する「箱」 |
| Flyway | DB マイグレーションツール | 箱の「形（テーブル構造）」を管理する |
| Spring Data JPA | データアクセスライブラリ | 箱の「中身（データ）」を読み書きする |
| ktlint | Kotlin の Linter/Formatter | Kotlin コードの「見た目」を整える（DB とは無関係） |
| Gradle | ビルドツール | ビルド・依存関係の解決・アプリの起動をする係 |
| mise | ツールバージョン管理 | JDK / Gradle などのバージョンを揃える係 |

## マイグレーションとは

DB のテーブル構造は開発が進むと変わっていく（例: users テーブルに email カラムを追加）。
これを手作業で `ALTER TABLE` すると、以下の問題が起きる。

- 自分の DB では変更したが、チームメンバーの DB に反映されない
- 本番環境で打ち忘れる
- 「今この DB はどの状態か」が誰にも分からなくなる

**Flyway を使うと**、変更を連番付き SQL ファイルとして残す:

```
src/main/resources/db/migration/
├── V1__create_users_table.sql
├── V2__add_email_to_users.sql
└── V3__create_posts_table.sql
```

Flyway は `flyway_schema_history` テーブルで「どこまで適用済みか」を記録し、
**未適用の SQL だけを自動で流す**。これにより:

- 誰の DB でも `V1 → V2 → V3` と同じ手順で同じ状態になる
- 変更履歴が Git に残り、コードと一緒にレビューできる

## 「ライブラリ」と「プラグイン」／「実行時」と「ビルド時」

同じ `build.gradle.kts` に書くものでも、動くタイミングと役割が違う。

### Flyway（今回はライブラリとして使う）

- `build.gradle.kts` に依存関係として書く JVM ライブラリ。
- Spring Boot がクラスパスに Flyway を見つけると、**アプリ起動時（ランタイム）に自動でマイグレーションを実行**する。
- そのため `./gradlew flywayMigrate` のようなコマンドは叩かなくてよい。
  `./gradlew bootRun` でアプリを起動した瞬間に Flyway が動く。
- イメージ: Gradle が Flyway という荷物をアプリに積み、**起動したアプリの中で Flyway が動く**。

> 補足: Flyway には Gradle プラグイン版 / CLI 版 / Docker 版もあるが、エンジンは同じで「入口」が違うだけ。今回はライブラリ版を使う。

### ktlint（プラグインとして使う）

- Gradle プラグイン。`./gradlew ktlintCheck` / `./gradlew ktlintFormat` のように
  **ビルド時・手動で動く**。アプリの動作とは無関係。

### まとめ

| | Flyway（今回） | ktlint |
|---|---|---|
| build.gradle での書き方 | 依存関係（ライブラリ） | プラグイン |
| いつ動く | アプリの**実行時**（ランタイム） | **ビルド時 / 手動** |
| 位置づけ | アプリの一部 | 開発ツール |
