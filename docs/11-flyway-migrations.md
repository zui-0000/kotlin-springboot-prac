# Flyway マイグレーション（テーブル定義の管理）

> テーブル定義を Flyway でどう管理するか。ファイル命名規約と、
> 実務向けのタイムスタンプ方式について。

## ファイル命名規約

Flyway のマイグレーションファイル名は規約で決まっている（守らないと認識されない）。

```
V   20260704105007   __   create_messages_table   .sql
│   │                │    │                        │
│   │                │    │                        └ 拡張子
│   │                │    └ 説明（人間用・自由）
│   │                └ 区切り（★アンダースコア2個★）
│   └ バージョン（適用順を決める）
└ プレフィックス（V = Versioned migration）
```

| 部品 | 意味 |
|------|------|
| `V` | バージョン付きマイグレーションの印 |
| バージョン | 適用する順番。数字（連番 or タイムスタンプ） |
| `__` | 区切り。**アンダースコア2個**（1個だと認識されない） |
| 説明 | 何をするファイルか。自由に書ける |
| `.sql` | 拡張子 |

## なぜ1つの DDL ファイルにまとめないのか

テーブル定義の管理には2つの思想がある。

| 方式 | 内容 |
|------|------|
| 状態ベース (state-based) | 1ファイルに「あるべき最終形」を宣言 |
| **マイグレーションベース (migration-based)** | **変更を1個ずつ順番に積み上げる** ← Flyway |

Flyway は後者。理由は「すでに稼働中の DB に差分をどう当てるか」問題を解決するため。

例: messages に title を追加したいとき——
- 状態ベースだと `CREATE TABLE` を書き換える → 既存 DB には既にテーブルがあり適用できない。
- マイグレーションベースなら新ファイルを足す:
  ```
  V..._create_messages_table.sql   （適用済み・触らない）
  V..._add_title_to_messages.sql   （新規）→ ALTER TABLE messages ADD COLUMN title ...
  ```
  Flyway は `flyway_schema_history` で適用済みを記録し、未適用分だけを流す。

## 黄金ルール：適用済みマイグレーションは編集しない

一度適用したファイルは二度と書き換えない。変更したければ**新しいファイルを足す**。

Flyway は各ファイルのチェックサムも記録しており、適用済みファイルを書き換えると
「中身が変わった」と検知して起動を止める（履歴改ざん防止の安全装置）。

## 連番の弱点と、タイムスタンプ方式（本プロジェクトで採用）

### 問題：連番(V1, V2...)は並行開発で衝突する

複数人が同時に「次は V3」と作ると、マージ時に V3 が重複する。

### 解決：バージョンをタイムスタンプにする

```
V20260704105007__create_messages_table.sql
 └────┬────┘
  YYYYMMDDHHMMSS（作成時刻）
```

- 同じ秒に2人が作ることはまずない → ファイル名の衝突が起きない。
- 作成時刻順に自然に並ぶ。
- TS の Prisma がマイグレーションフォルダをタイムスタンプで作るのと同じ発想。

### 必要な設定：out-of-order

タイムスタンプ方式では「後から古い時刻のマイグレーションがマージされる」ことがある。
Flyway はデフォルトで番号順にしか適用しないため、これを許可する設定が要る。

```yaml
# application.yml
spring:
  flyway:
    out-of-order: true
```

### 本質：変更の「順序」は人間が考えるもの

タイムスタンプはファイル名の衝突を消すだけ。
「どの変更をどの順で当てるか」という本質的な調整は、どの方式でも残る（残るべき）。
スキーマ変更には依存関係があるため（テーブルを作る前にカラムは足せない）。

## 新しいマイグレーションの追加手順

1. `V<タイムスタンプ>__<説明>.sql` を作る（例: `V20260705093000__add_title_to_messages.sql`）。
2. 中に DDL を書く（`ALTER TABLE ...` など）。
3. 必要なら Exposed の `Table` 定義も合わせて更新（[09-exposed.md](./09-exposed.md)）。
4. アプリを起動すると Flyway が未適用分を自動適用する。

> 適用済みの DB を作り直したいとき（ローカル開発のみ）:
> `docker compose down -v && docker compose up -d` で DB をまっさらにして再適用。
