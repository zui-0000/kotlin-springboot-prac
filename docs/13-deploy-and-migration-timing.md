# デプロイとマイグレーションの実行タイミング

> `bootRun`（開発）と本番稼働の違い、マイグレーションをいつ流すか、のメモ。

## `bootRun`（dev）は「開発専用」コマンド

`mise run dev`（= `./gradlew bootRun`）は**ローカル開発でのみ使う**。本番では使わない。

アプリのライフサイクル:

```
【開発】  mise run dev (bootRun)   ← ローカルで動かす。開発専用
   ↓
【ビルド】 ./gradlew build          ← build/libs/xxx.jar が生成される
   ↓
【デプロイ】サーバー / コンテナに jar を配置
   ↓
【本番稼働】java -jar xxx.jar        ← ★本番はこれでアプリを動かす★
```

- 本番でアプリを動かすのは **`java -jar`**。`bootRun` ではない。
- 「デプロイに dev が必要」ではない。dev は本番工程に含まれない。

## マイグレーションは「アプリ起動時」に紐づく

Flyway の自動マイグレーションは **アプリが起動する時**に走る。
`bootRun`（dev）でも `java -jar`（本番）でも、どちらも「アプリ起動」なので、
（有効なら）同じように Flyway が走る。

→ マイグレーションは `bootRun` に紐づくのではなく、**「アプリ起動」に紐づく**。

## 本番で自動マイグレーションをオフにする理由

本番ではアプリを**複数インスタンス**で動かすことが多い（負荷分散・冗長化）。

```
本番環境
  ├─ インスタンス1 起動 ┐
  ├─ インスタンス2 起動 ├─ 全部が同時にマイグレーションしようとして競合！
  └─ インスタンス3 起動 ┘
```

そのため本番の定石はこう:

```
【デプロイ手順】
① マイグレーションだけ、先に1回流す        ← 独立コマンド（Flyway CLI / Gradle プラグイン）
② その後、アプリ群を起動（起動時の自動マイグレーションはオフ）
```

`spring.flyway.enabled: false` にして自動マイグレーションを止め、
①を専用ステップで実行する。「本番でマイグレーションを立てたいとき」とは、この①のこと。

## 本プロジェクトとの対応

| 環境 | マイグレーション | アプリ起動 |
|------|----------------|-----------|
| ローカル（本プロジェクト） | 起動時に自動（1台なので楽） | `mise run dev` |
| 本番（複数インスタンス想定） | `flyway.enabled: false` にして独立実行 | `java -jar`（自動はオフ） |

- 本プロジェクトの `mise run db-migrate`（独立コマンド）は、まさに本番手順①に相当する。
  （[11-flyway-migrations.md](./11-flyway-migrations.md) の「マイグレーションの適用方法」参照）
- 学習用で1インスタンスのため「起動時自動」でシンプルに運用している。
  方針が違うだけで、どちらも正しい。

## まとめ

| 疑問 | 答え |
|------|------|
| build してデプロイして終わり？ | ほぼそう。ただしデプロイ後に `java -jar` で「動かす」工程がある |
| dev しないとデプロイ完了しない？ | No。dev(bootRun) は開発専用。本番は `java -jar` |
| 本番でマイグレーションを立てるとき？ | 複数インスタンス構成で、アプリ起動と切り離して専用ステップで流したいとき |
